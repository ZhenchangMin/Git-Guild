# ADR-003：采用 REST + Webhook + 应用内事件的集成方式

## 状态
已接受

## 背景

Git Guild 既需要支持前后端交互，也需要接收外部代码托管底座的事件，同时还要在单体内部解耦审核、通知和成长更新流程。团队需要在简单可实现与适度解耦之间找到平衡。

## 决策

我们决定采用以下集成方式：

- 前后端之间使用 REST API + JSON
- 与 Gitea 之间使用 HTTP API 与 Webhook
- 单体内部通过应用内事件处理通知、成长等后续动作

## 理由

- REST API 对前端最直观，调试成本最低
- Webhook 是接收 PR、Issue、仓库状态变化的自然方式
- 应用内事件能够在不引入消息队列的情况下实现适度解耦
- 对课程项目而言，没有必要提前引入 MQ、gRPC 或 API 网关

## 后果

- 正面影响：实现简单、链路清晰、便于排查和联调
- 负面影响：应用内事件天然受限于单体进程，后续若拆服务需要重构部分事件流
- 需要关注的风险：Webhook 重复投递、乱序和失败重试机制需要实现阶段补强

## AI 辅助记录

- AI 初稿内容摘要：AI 曾建议使用 gRPC、RabbitMQ 或 NATS 作为服务间通信与事件总线
- 人工修订内容：团队将通信方式收敛为 REST、Webhook 和应用内事件
- 修订理由：AI 初稿默认存在多服务部署前提，不适合当前单体架构和课程周期

## AI 初稿与人工修订对比

```diff
- 内部服务使用 gRPC 通信
- 事件通过 RabbitMQ / NATS 进行异步广播
+ 前后端统一使用 REST + JSON
+ 外部托管底座通过 Webhook 推送事件
+ 单体内部使用应用内事件处理通知与成长更新
```
