# ADR-002：采用平台自有账号体系并集成 Gitea 作为外部代码托管底座

## 状态
已接受

## 背景

团队讨论后认为，Git Guild 不应被绑定为 GitHub 附属层。平台需要拥有独立的用户账号、角色与业务规则，同时又不能在课程周期内自研完整的 Git 仓库、Issue、PR 与 Webhook 底座。

## 决策

我们决定：

- 使用平台自有账号体系完成注册、登录和角色管理
- 集成 Gitea 作为外部代码托管底座
- Git Guild 自身只负责上层任务协作、推荐匹配、审核反馈与成长激励

## 理由

- 平台自有账号体系可以避免产品边界被 GitHub 登录方式限制
- 自研代码托管底座成本过高，会偏离课程项目重点
- 相比 GitLab，Gitea 更轻量；相比 GitHub 远程依赖，Gitea 更适合本地或课程环境自托管
- Gitea 对仓库、Issue、PR、Webhook 的支持已经满足项目需求

## 后果

- 正面影响：平台边界更清晰，底层托管能力复用成熟产品，开发重心集中在核心业务
- 负面影响：需要处理平台账号与 Gitea 侧身份、权限、资源同步之间的映射问题
- 需要关注的风险：Gitea API、Webhook 顺序和异常重试需要在实现时重点考虑

## AI 辅助记录

- AI 初稿内容摘要：AI 初稿混合推荐 GitHub OAuth、GitHub API、GitLab API 和 Gitea API，未明确平台边界
- 人工修订内容：团队明确平台应采用自有账号体系，并将外部代码托管底座统一收敛为 Gitea
- 修订理由：AI 初稿把“登录方式”“代码托管底座”和“平台核心边界”混在了一起，需要人工拆清

## AI 初稿与人工修订对比

```diff
- 采用 GitHub OAuth 登录，并可先通过 GitHub Webhook 实现上层业务
- Git 集成可选 GitLab/GitHub API 或 Gitea API
+ 采用平台自有账号体系，不依赖 GitHub OAuth
+ 统一选择 Gitea 作为外部代码托管底座
+ 平台核心业务集中在任务、推荐、审核、成长，不重做底层 Git 托管
```
