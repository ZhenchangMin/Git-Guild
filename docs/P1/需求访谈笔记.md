# 需求访谈笔记

## 访谈对象1：计算机专业学生

### 访谈内容

- Q1：你目前的学习/开发状态是什么？最近三个月主要做哪些项目类型？

A：目前主要还是学生身份，最近三个月主要在做课程项目，也会做一点科研项目。大概课程项目占七成，科研项目三成左右。技术栈主要是 Vue 和 Spring Boot，有一些简单的前后端协作经验。

- Q2：回忆一次你“想参与但没有参与”或“中途放弃”的开源协作经历，发生了什么？

A：我参与的《Web开发》课程项目没写完，因为其**架构不清晰**，太麻烦了。

- Q3：你在任务列表里决定“接不接”的前三重要的信息是什么？什么样的系统推荐值得你的信任？

A：我在任务列表里决定接不接，最看重三点：第一是**难度是否清晰**，能不能判断适不适合我；第二是**说明是否完整**，有没有具体步骤；第三是**预计完成时间**，不希望周期太长。
相比按历史行为或者推荐算法，我更信任按**新手路径和难度分层**来推荐的方式，这样更有安全感。

- Q4：接到任务后，你接取后最怕什么因素不清晰？你最需要平台提供哪三类支持？

A：我最怕不知道做到**什么程度**才算完成。我最需要的平台支持有三类：第一是完整的**流程模板**；第二是有**示例**，比如参考 PR 或示例代码；第三是清晰的**提交流程说明**。

- Q5：你认为什么样的“完成标准”才足够清晰？

A：我觉得好的完成标准应该包含几个方面：要实现哪些**功能点**，是否需要写**测试**，用什么**格式**提交代码，以及是否需要**补充文档**。
如果可以的话，最好做成 **checklist** 的形式，一条一条列出来，我可以对照完成。

- Q6：提交后你最希望得到什么反馈？你可接受的反馈时限是多久？

A：提交之后我最希望得到的是比较**具体**的反馈，最好能直接告诉我哪里需要改，怎么改，而不是只说通过或者不通过。
时间上我可以接受在**三天之内**有反馈。

- Q7：任务状态变化时，哪些事件必须通知你？你更偏好什么频率和渠道？

A：任务状态变化中，我觉得必须通知的包括：任务**是否被接收**、**PR** 是否被 review、是否需要**修改**。
通知方式上我更倾向于**站内加邮件**，同时希望有一个**汇总机制**，比如每天统一通知一次，避免太频繁打扰。

- Q8：哪些因素最能让你持续参与？

A：让我持续参与的主要因素有三个：第一是能不能**真正学到东西**；第二是这些经历能不能**写进简历**；第三是能不能得到**维护者的认可**。像等级、徽章这些是加分项，但不是最核心的动力。

- Q9：你对“冒险家公会”的游戏化设计和“连续贡献机制”的接受边界是什么？

A：我对游戏化机制是可以接受的，比如等级、徽章这些会有一点激励作用。但如果设计得太强，比如连续中断就清零，会有比较大的**压力**，反而容易放弃。整体来说可以有游戏化，但**不要太强制**。

### 访谈总结

- A. 受访者画像：
	1) 软件工程专业学生；近三个月课程项目约占 70%、科研项目约占 30%；
    2) 具备基础前后端协作经验，主要技术栈为 Vue 与 Spring Boot。
- B. 关键痛点：
	1) 项目**架构不清晰**，导致参与中途放弃，前期理解成本高。
	2) 接取任务时常遇到“难度、说明、预计时长”信息不充分，影响决策信心。
	3) 接取后最怕**完成标准边界不清**，不知道做到什么程度才算完成。
- C. 期望支持：
	1) 提供**完整流程模板**，明确从接取、开发到提交的步骤。
	2) 提供**可参考的示例**（示例 PR、示例代码、示例文档）。
	3) 提供清晰提交流程与验收 **checklist**（功能点、测试、提交格式、文档补充）。
- D. 反馈与通知规则（SLA + 渠道 + 频率）;
	1) 反馈 SLA：提交后 **3 天内**给出首轮反馈；反馈需具体到“哪里改、怎么改”。
	2) 通知渠道：站内通知 + 邮件。
	3) 通知频率：**关键状态**即时触发（是否接收、PR 是否被 review、是否需修改），并支持每日 1 次**汇总通知**以降低打扰。
- E. 激励偏好与游戏化边界;
	1) 核心激励偏好：能学到东西、可写入简历、得到维护者认可。
	2) 游戏化接受度：可接受等级/徽章等**轻量正向**激励。
	3) 边界约束：不接受“连续中断即清零”等**强惩罚机制**，避免压力导致流失。
- F. 可直接写入 SRS 的需求条目。
	1) 系统应在任务卡片中展示任务难度、任务说明完整度与预计完成时长。
	2) 系统应提供面向**新手路径**与**难度分层**的任务推荐，并在推荐结果中标注推荐依据。
	3) 系统应提供标准化**完成标准模板**，最好支持 checklist 视图。
	4) 系统应提供**流程指引**与**示例资源**（示例 PR、示例代码），供贡献者在提交前自检。
	5) 系统应在提交后 72 小时内触发首轮评审反馈提醒；超时时向维护者发送升级通知。
	6) 系统应支持任务状态**关键事件通知**（接收结果、PR review、需修改）并提供站内通知与邮件双通道。
	7) 系统应提供通知汇总策略，支持**每日一次**摘要推送以减少高频打扰。
	8) 系统应提供轻量游戏化激励配置，并默认禁用“中断清零”类惩罚规则。

## 访谈对象2：开源项目维护者

### 访谈内容

- Q1：你目前的学习/开发状态是什么？最近三个月主要做哪些项目类型？

A：我是一名开源项目维护者，最近三个月主要在维护和更新我的开源项目，同时也参与了一些社区活动。我的技术栈主要是Go和Rust。

- Q2：回忆一次你“想参与但没有参与”或“中途放弃”的开源协作经历，发生了什么？

A：有一次我想参与一个开源项目，但在**环境搭建**阶段遇到了很多问题，最终放弃了。主要是因为**文档不够详细**，缺乏新手友好的指导。

- Q3：你在任务列表里决定“接不接”的前三重要的信息是什么？什么样的系统推荐值得你的信任？

A：我决定接不接一个任务，主要看**任务难度**、**技术栈**和**预计时长**。我更信任那些按**技术栈匹配**和按**历史行为**推荐的系统，因为它们更符合我的**兴趣和能力**。

- Q4：接到任务后，你接取后最怕什么因素不清晰？你最需要平台提供哪三类支持？

A：我最怕**完成标准不清楚**。我最需要平台提供**完成标准模板**和**参考示例**。

- Q5：你认为什么样的“完成标准”才足够清晰？

A：我认为完成标准应该包含**功能点**、**测试要求**和**代码规范**，这样才能让我清楚地知道需要完成什么。

- Q6：提交后你最希望得到什么反馈？你可接受的反馈时限是多久？

A：我最希望得到**具体到可执行修改建议**的反馈。对于反馈时限，我希望能在**24小时内**得到回复，这样可以及时调整我的工作。

- Q7：任务状态变化时，哪些事件必须通知你？你更偏好什么频率和渠道？

A：我觉得**任务状态变化**时通知我，特别是**审核结果**和**任务更新**。我更偏好通过**邮件**和**平台内通知**接收这些信息，这样可以确保我不会错过重要的更新。

- Q8：哪些因素最能让你持续参与？

A：我觉得**明确的学习路径**和**成就感**是最能让我持续参与的因素。明确的学习路径可以帮助我看到自己的进步，而成就感则是对我努力的认可。

- Q9：你对“冒险家公会”的游戏化设计和“连续贡献机制”的接受边界是什么？

A：我觉得“冒险家公会”的游戏化设计很有趣，但我不喜欢**中断清零**的机制，因为这会让我感到压力。对于“连续贡献机制”，我觉得只要它能提供**适当的激励**而不是惩罚，我是可以接受的。

- M1：您每次发布任务时最耗时的步骤是什么？

A：我觉得最耗时的步骤是**编写清晰的任务描述**和**制定合理的完成标准**，因为这需要我仔细考虑任务的每一个细节，以确保参与者能够理解并顺利完成任务。

- M2：多人抢同一任务时，平台应如何分配与冲突处理才公平？

A：我觉得平台可以采用**先到先得**的方式来分配任务，但同时也应该提供一个**等待列表**，让其他人有机会参与。如果出现冲突，平台可以通过**轮流分配**的方式来处理，这样可以确保公平性。

- M3：你可接受的审核工作量上限是什么（每天/每周）？

A：我觉得每天审核**不超过5个任务**是可以接受的，这样可以保证我有足够的时间来仔细审核每个任务，同时也不会感到过于压力。如果任务量较大，我希望平台能提供一些**自动化工具**来帮助我提高审核效率。

### 访谈总结

- A. 受访者画像：
	1) 开源项目维护者；
    2) 近三个月以维护与更新开源项目、参与社区活动为主。
- B. 关键痛点：
	1) 新贡献者在**环境搭建**阶段流失，核心原因是文档不够详细、缺少新手友好引导。
	2) 接取任务后**完成标准不清晰**，导致返工与沟通成本高。
- C. 期望支持：
	1) 平台提供**完成标准模板**与参考示例（覆盖功能点、测试要求、代码规范）。
	2) 推荐系统基于**技术栈匹**配与**历史行为**推荐，提高匹配可信度。
	3) 提供**自动化辅助审核工具**，降低人工审核负担。
- D. 反馈与通知规则（SLA + 渠道 + 频率）：
	1) 反馈 SLA：提交后 **24 小时内**给出首轮反馈，反馈内容需可执行到修改项级别。
	2) 通知渠道：邮件 + 平台内通知。
	3) 通知频率：审核结果、任务更新等关键事件即时通知。
- E. 激励偏好与游戏化边界;
	1) 持续参与驱动：明确学习路径 + 成就感。
	2) 可接受边界：接受适度**正向激励**的连续贡献机制。
	3) 不可接受边界：反对“中断即清零”的**强惩罚机制**。
- F. 可直接写入 SRS 的需求条目。
	1) 系统应提供标准化任务模板，至少包含任务背景、功能点、验收标准、测试要求与代码规范字段。
	2) 系统应在任务详情页展示难度、技术栈、预计时长三个核心决策信息。
	3) 系统应基于贡献者**技术栈**与**历史行为**进行个性化推荐，并提供推荐理由说明。
	4) 系统应在提交后 **24 小时内**触发首轮审核反馈提醒；若超时，自动升级通知给维护者。
	5) 系统应提供任务分配冲突处理机制，支持**先到先得**等分配策略。
	6) 系统应支持维护者设置**审核负载上限**（例如每日不超过 5 个任务），超限后自动延后分配或推荐协审。
