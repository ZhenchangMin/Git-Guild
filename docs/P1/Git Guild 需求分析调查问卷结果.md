## 问卷结果分析

本次问卷共收集有效填写 27 份。下文围绕受访者画像、核心痛点、功能期待、游戏化态度、维护者视角五个维度展开分析总结。

### 一、受访者画像：以“新手 + 课程级经验 + 零开源经历”为主

- **身份分布**（Q1）：77.78% 为开源新手/初级开发者，有一定经验者仅 11.11%，项目维护者仅 1 人。
- **编程经验**（Q2）：66.67% 仅能完成课程作业或小项目，14.81% 刚开始学习编程，合计逾 80% 位于"初级"区间。
- **开源经历**（Q3）：44.44% 从未参与过开源，25.93% 想过但从未开始，仅 29.63% 有过实际贡献。

**结论**：样本高度集中于 Git Guild 的核心目标用户："想入门却未入门"的开源新手。因此后续分析结果可作为新手向产品设计的主要参考。
但也需注意维护者视角样本偏少，项目维护者需求将在访谈环节进一步补充。

### 二、参与开源的核心痛点：入口缺失 + 流程陌生 + 动力不足

Q4 显示最突出的三大障碍依次为：

1. **找不到适合入门的任务**（85.19%，首位痛点）
2. **不清楚贡献流程 / 缺少时间或持续动力**（59.26%）
3. **难以理解项目代码或规范**（48.15%）

结合 Q5，81.48% 的受访者"非常愿意"或"比较愿意"尝试专门面向新手的开源任务平台

**结论**：用户的首要痛点是**任务可得性**而非能力不足，这直接验证了 Git Guild 任务板和推荐机制的立项假设。
同时"贡献流程不清"和"动力不足"两项说明必须 Git Guild 同时解决**流程引导**与**持续性激励**。

### 三、任务筛选与支持需求

**筛选维度**（Q6）：技术栈（74.07%）、任务难度（66.67%）、详细说明（59.26%）、是否适合新手（55.56%）为前四项，奖励和审核标准反而权重较低，说明用户更关心"我能不能做、怎么做"。

**期望支持**（Q7）：清晰的任务说明和完成标准（85.19%）、贡献流程指引（66.67%）、类似任务示例（66.67%）、与发布者沟通渠道（62.96%）均为高频选项，"不需要额外帮助"为 0。

**反馈偏好**（Q10）：81.48% 希望获得"详细 Review 意见"或"附带学习建议"。

**结论**：任务详情页和贡献流程引导应作为 MVP 必做项。
任务卡片的核心展示字段应是**技术栈、难度、是否适合新手、说明完整度**，而非奖励金额。

### 四、技术接入偏好：GitHub 登录 + 邮件通知为主

- **登录方式**（Q8）：GitHub 账号（62.96%）居首，手机号（48.15%）、邮箱（40.74%）并列其次。GitHub OAuth 是新手群体已有的身份体系，应作为首发登录方案。
- **通知渠道**（Q9）：邮件提醒（85.19%）显著高于站内通知（51.85%），仅 7.41% 选择"不需要及时通知"。邮件通知在任务状态变更场景中应作为默认开启项，而非附加。

### 五、游戏化与激励机制：认可与成长价值 > 虚拟排名

Q11 的持续参与意愿因素前三：

| 排名 | 因素 | 比例 |
| --- | --- | --- |
| 1 | 实际技能提升 | 74.07% |
| 2 | 可展示的经历或履历价值 | 66.67% |
| 3 | 来自维护者的认可 | 62.96% |

**冒险家公会设定**（Q12）：70.37% 认为有激励作用（很有 22.22% + 有一定 48.15%），但 14.81% 担心"不够严肃"。

**打卡/连续贡献**（Q13）：需要仅 33.33%，不需要反而占 44.44%。

**结论**：
- 游戏化元素应**服务于真实成长价值**（技能、履历、认可），而非纯虚拟激励。XP/等级是可接受的"副产品"，排行榜热度较低，打卡机制用户普遍不买账。
- 冒险家公会概念整体正向，但存在"不够严肃"的担忧；视觉和文案需在"趣味"与"专业"之间谨慎平衡，避免过度玩具化。
- 徽章、排行榜、打卡机制**不应进入 MVP**，应优先打磨贡献记录与维护者认可机制。

### 六、维护者视角：任务质量与协同效率是核心

尽管维护者样本量小，Q14、Q15 的结果仍指向明确方向：

- **最希望平台解决的问题**（Q14）：快速编写清晰任务描述（88.89%）位居首位，合理设置难度奖励（66.67%）、管理冲突（59.26%）、高效审核（55.56%）、保持沟通（55.56%）紧随其后。
- **留在平台的动力**（Q15）："找到参与者帮助完善项目"以 77.78% 遥遥领先，打赏与新手夸奖均为 48.15%。

**结论**：维护者更看重**协作效率与实际产出**，而非金钱激励。平台应优先提供任务模板/编写辅助、接取冲突管理、审核工作流等工具。

### 七、核心功能期待：新手推荐 + 学习引导优先

Q16 功能优先级排序：

1. 新手任务推荐（77.78%）
2. 学习引导/教程支持（70.37%）
3. 审核反馈系统（40.74%）
4. 成长系统 / 维护者发布任务（各 29.63%）
5. 排行榜（7.41%）

**结论**：用户最需要的是"找到任务 + 学会怎么做"。排行榜用户冷感明显，确认其不应占据 MVP 资源。

### 八、对 Git Guild 产品设计的总体启示

1. **MVP 必做**：GitHub OAuth 登录、新手任务推荐、任务详情页（含技术栈/难度/新手标签/详细说明）、贡献流程引导、邮件+站内通知、XP/等级的轻量成长系统。
2. **MVP 不做**：排行榜、打卡/连续贡献、复杂的打赏经济。
3. **风险与平衡**：冒险家公会的视觉语言需在"游戏化"与"专业感"之间找到平衡点；成长体系应始终锚定"真实技能 + 履历价值"，避免沦为虚拟数字游戏。
4. **后续调研**：维护者样本过少（仅 1 人），其需求需通过访谈深入补充，尤其是任务冲突管理、审核流与沟通机制的具体痛点。
