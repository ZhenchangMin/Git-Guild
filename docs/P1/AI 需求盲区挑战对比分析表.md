# AI需求盲区挑战对比分析表

| 对比维度            | AI 生成的需求                                                                                                                                                                                                                                   | 人工调研发掘的需求                                                                                                                                                                                               | 分析                                                                                                                                                               |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| AI 正确覆盖的需求   | 1) 核心功能闭环：仓库导入 -> 基于 Issue 发布任务 -> 接取 -> 提交 PR/成果 -> 审核 -> 成长记录。<br />2) 关键角色与职责：冒险家、委托人、管理员，以及 GitHub/Gitea 外部系统。<br />3) 关键筛选维度：技术栈、难度、预计时长。                      | 1) 访谈提纲、用例、用户故事等共同确认上述主链路是课程 MVP 主体。<br />2) 问卷结果显示“任务推荐”“学习引导”“审核反馈”是高优先需求。<br />3) 访谈多名受访者一致强调技术栈匹配、难度和说明完整度。             | AI 在“主要流程建模”和“结构化输出”方面有效，适合快速搭建初稿**骨架**。它对显性、通用、流程型需求覆盖较好。                                                |
| AI 遗漏的需求       | 1) 反馈机制细化：24h/72h/1周分层期望。<br />2) 通知策略细化：**关键事件即时** + **每日汇总**的可配置机制。<br />3) 维护者**审核上限**：如每日 <= 5个任务。<br />4) 减少强惩罚机制：连续贡献机制应可选，默认避免“中断清零”。 | 1) 访谈对象多次提到“完成标准边界不清”“希望具体可执行反馈”“不希望长期无反馈”。<br />2) 问卷显示邮件通知偏好明显高于站内，且希望有即时+汇总机制。<br />3) 维护者访谈明确提出任务描述编写成本和审核负载上限。 | 为什么 AI 会遗漏：<br />1) Prompt 前期偏“格式完整”，对细化角色约束不精确。<br />2) AI 更擅长功能面描述，对运营规则、工作负载和行为激励这类“隐性约束”捕捉不足。 |
| AI 捏造的需求       | 1) 过早引入完整商业化扩展：复杂社区运营、大而全推荐体系、重度游戏化内容。<br />2) 以排行榜等虚拟激励作为优先项。                                                                                                                                | 1) 问卷显示用户核心驱动是**技能提升**、**可展示经历**、维护者认可，不是排行榜。<br />2) 问卷中“是否需要连续贡献机制”总体**不支持强推动**。                                                   | 这些需求并非“完全不存在”，但不属于当前高优先级或必须项。AI 的问题是**时机与优先级捏造**，把后续功能提前成 MVP 必做。                                       |
| AI 表述有歧义的需求 | 1) 验收标准里出现“及时反馈”“清晰标准”等不可测词。<br />2) 角色边界不严：谁可审核、谁可修改、何时可见操作入口有时未明确。<br />3) 状态定义不严：任务在待审核/退回/完成的触发条件与异常回滚条件不总是一致。                                   | 1) 人工版本用户故事补充了 Given-When-Then 的异常路径和可观察结果。<br />2) 用例和用户故事修订中增加了权限不足、导入失败、PR 关联失败、任务被占用等边界条件。                                                     | 歧义主要在“不可测试”和“不可执行”：<br />1) 未量化验收标准。<br />2) 未明确角色与状态流转。                                                                     |
