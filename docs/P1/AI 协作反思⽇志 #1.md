## 2. AI 助力分析（Q1）
本阶段 AI 最大的贡献是：快速产出结构化初稿，尤其在 SRS、用例图和用户故事骨架搭建上显著降低了从“空白页”到“可审查版本”的时间成本，同时帮助团队统一了“冒险家/委托人/Quest/XP”等核心术语表达。
估计节省时间：约 6~8 小时。

## 3. AI 误导分析（Q2）
发现的 AI 输出问题：
- 问题1：偏向正常流程，容易遗漏异常流和边界场景，如仓库导入失败、无权限访问、任务已被他人接取、PR 关联失败等。如何发现：将 AI 版本与访谈笔记、问卷结果和核心业务链路逐项对照时发现缺口。如何修正：在详细用例和用户故事中补充异常路径，并明确状态流转条件。
- 问题2：部分输出会把课程展示版 MVP 扩展成“完整商业平台”，混入复杂推荐、社区运营等超出当前阶段的内容。如何发现：与课程要求、团队工期和调研优先级对齐时发现范围偏大。如何修正：人工收缩为“仓库接入—任务发布/接取—成果提交—审核反馈—成长记录”的核心闭环，其余功能降为后续扩展。
- 问题3：部分验收标准虽然形式上使用了 Given-When-Then，但仍存在状态不清、角色边界不严谨、结果不可直接测试的问题。如何发现：在 3.3 对比分析和逐条审读用户故事时发现。如何修正：统一使用明确状态名、前置条件、异常条件和可观察结果，删除“方便”“及时”等模糊表述。

## 4. Prompt 改进计划（Q3）
本阶段使用的 Prompt 策略存在的问题：前期 Prompt 更强调“格式完整”和“一次性生成”，但对“仅基于已有调研证据输出”“必须覆盖异常流”“限定课程 MVP 范围”等约束不够强；另外，Prompt 日志留存也不够完整，部分问卷与修订过程缺少记录。
下阶段计划采用的改进措施：采用“证据约束 + 范围约束 + 反例约束”的 Prompt 模板，先给出问卷/访谈结论，再要求 AI 只输出可追溯需求；每项至少补 1 条异常路径；输出待人工确认项；所有迭代 Prompt 当天归档。

## 5. 本阶段的核心工程判断
（记录一个 AI 无法替你做出的关键决策）
决策内容：将 P1 的核心闭环确定为“项目/仓库接入—基于 Issue 发布任务—任务接取—成果提交—审核反馈—成长记录”，并把复杂推荐、强惩罚式游戏化和社区扩展功能降为非 MVP。
为什么 AI 无法做这个决策：这个取舍依赖课程展示目标、团队工期、调研样本权重和实现成本的综合判断，需要团队结合现实资源做权衡，AI 可以提供备选方案，但不能替代最终的工程取舍。
