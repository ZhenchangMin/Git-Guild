## 4. 统一 API 约束

| 约束项       | 规范说明                                                                 |
| ------------ | ------------------------------------------------------------------------ |
| 基础路径     | 所有业务接口统一使用 `/api/v1`                                           |
| 数据格式     | 请求体和响应体统一使用 JSON                                              |
| 认证方式     | 登录态通过 `Authorization: Bearer <JWT_TOKEN>` 传递                      |
| 响应结构     | 成功和失败均返回 `code`、`message`、`timestamp`、`traceId`               |
| 错误处理     | 每个模块文档独立列出业务错误码和 HTTP 状态码                             |
| 权限校验     | 角色权限必须由后端校验，前端页面控制不能作为安全边界                     |
| 分页参数     | 列表接口统一使用 `page` 和 `size`，页码从 `1` 开始                       |
| 幂等处理     | Webhook、同步、审核等重复调用风险较高的接口必须在服务端做重复处理保护    |

## 5. AI 生成 API 草案审查记录

| 编号 | 审查维度     | AI 草案问题                                                         | 人工修正结果                                                       |
| ---- | ------------ | ------------------------------------------------------------------- | ------------------------------------------------------------------ |
| 1    | 接口命名     | 将任务接口混写为 `/tasks`、`/orders`、`/quests`，领域语言不一致     | 统一使用 `/quests` 表达悬赏任务，避免接口语义漂移                  |
| 2    | 错误处理     | 只给出成功响应，缺少失败响应和错误码                                | 每份 API 文档补充业务错误码、HTTP 状态码和典型失败响应             |
| 3    | 参数校验     | 未说明必填字段、枚举值和分页边界                                    | 所有请求参数表增加类型、是否必填、枚举值和默认值                   |
| 4    | 安全问题     | 接取任务、审核任务等接口在草案中允许前端传 `userId`                 | 接取人、审核人统一从 JWT 解析，前端不传可信身份字段                |
| 5    | 权限边界     | 未区分初学者、维护者、管理员的可调用接口                            | 在接口说明中明确角色权限，例如发布任务、管理员审核、维护者审核     |
| 6    | 状态流转     | 草案允许直接把任务改为任意状态                                      | 在接口说明中固定状态流转，例如任务发布后先进入管理员审核           |
| 7    | 外部平台耦合 | 把 GitHub OAuth 作为默认登录方式                                    | 改为平台自有账号体系；GitHub 只作为仓库导入和同步来源              |
| 8    | Webhook 安全 | Webhook 接口未考虑签名校验                                          | 代码支撑平台 API 增加 Webhook 签名、幂等和重复投递处理要求         |

## 6. 审查结论

AI 生成的 API 草案可以帮助快速铺开接口清单，但直接使用会产生命名不一致、权限不清晰和错误处理不足的问题。最终版本以 P1 需求和 P2 架构为准，保留 RESTful 风格，同时补足身份认证、角色权限、参数校验、错误码和外部代码托管安全约束。
