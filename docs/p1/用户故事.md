## 冒险家
1. 作为冒险家，我希望按难度、技术栈和标签筛选任务，以便找到与自己当前能力匹配的任务。  
   验收标准 1：Given 任务大厅中存在多个已发布任务，When 我选择 `Java`、`D/C 难度` 和 `新手友好` 作为筛选条件，Then 列表仅显示同时满足这三个条件的任务。  
   验收标准 2：Given 当前筛选结果不少于 2 条任务，When 我按 `XP 奖励从高到低` 排序，Then 列表顺序与各任务的奖励数值一致。  
   验收标准 3：Given 当前筛选条件下没有任何匹配任务，When 我执行筛选，Then 页面显示 `无匹配任务`，且不显示上一次筛选的旧结果。  

2. 作为冒险家，我希望查看任务详情及其关联仓库信息，以便判断自己是否能完成该任务。  
   验收标准 1：Given 某任务已发布且已关联仓库与 Issue，When 我打开任务详情，Then 页面显示任务标题、技术栈、预计工作量、完成标准、奖励和关联 Issue 标题。  
   验收标准 2：Given 任务关联的仓库已成功接入平台，When 我切换到仓库信息区域，Then 我可以查看该任务相关的分支、提交记录、文件和差异。  
   验收标准 3：Given 任务关联的仓库或 Issue 数据当前不可用，When 我打开任务详情，Then 页面显示 `仓库信息暂不可用` 或 `Issue 信息暂不可用`，且不显示接取按钮。  

3. 作为冒险家，我希望接取未分配的悬赏任务，以便开始完成真实项目中的协作工作。  
   验收标准 1：Given 某任务状态为 `待接取` 且当前没有接取人，When 我点击 `接取任务` 并确认，Then 任务状态变为 `进行中`，且接取人显示为我。  
   验收标准 2：Given 我已经成功接取某任务，When 我再次进入该任务页面，Then 页面显示该任务的完成标准、关联仓库信息和当前状态 `进行中`。  
   验收标准 3：Given 另一名冒险家已先一步接取该任务，When 我提交接取请求，Then 系统拒绝本次接取，任务状态保持不变，并显示当前接取人。  

4. 作为冒险家，我希望提交与任务关联的 Pull Request 或成果说明，以便让委托人进入审核流程。  
   验收标准 1：Given 我已接取某任务且该任务关联了平台内仓库，When 我提交包含 PR 链接和成果说明的成果，Then 任务状态变为 `待审核`。  
   验收标准 2：Given 我的成果已提交成功，When 我查看该任务的提交记录，Then 页面显示本次提交的 PR 标题、提交说明和提交时间。  
   验收标准 3：Given 我提交的内容缺少 PR 链接，或 PR 指向与任务无关的仓库，When 我提交成果，Then 系统拒绝提交，且任务状态保持 `进行中`。  

5. 作为冒险家，我希望查看逐项审核意见和成长结果，以便根据反馈改进并积累可展示的经历。  
   验收标准 1：Given 委托人已对我的成果作出 `请求修改` 结论，When 我打开审核结果页面，Then 页面显示任务状态 `已退回` 以及逐项修改意见。  
   验收标准 2：Given 委托人已通过我的成果，When 我打开已完成任务页面，Then 页面显示本次获得的 XP、当前等级进度和该任务的项目经历记录。  
   验收标准 3：Given 我的任务仍处于 `待审核` 状态，When 我打开审核结果页面，Then 页面显示 `待审核`，且 XP 数值保持不变。  

## 委托人
1. 作为委托人，我希望将 GitHub 项目仓库导入到平台，以便在平台内发布任务并跟踪协作过程。  
   验收标准 1：Given 我提供了可访问的 GitHub 仓库地址，When 我发起导入，Then 平台创建该项目并展示仓库的分支、提交记录、文件和差异。  
   验收标准 2：Given 导入的仓库中存在 Issue，When 导入完成，Then 这些 Issue 可在平台内被查看并用于后续创建任务。  
   验收标准 3：Given 外部平台接口不可用或仓库导入失败，When 我发起导入，Then 项目状态显示为 `导入失败`，且该仓库不会以可接取项目的状态出现在平台中。  

2. 作为委托人，我希望基于仓库中的 Issue 创建结构化悬赏任务，以便把零散问题转化为可分配、可跟踪的任务。  
   验收标准 1：Given 某个 Issue 处于开放状态，When 我从该 Issue 创建任务并填写难度、技术栈、预计工作量、完成标准和奖励，Then 任务发布成功且与该 Issue 建立关联。  
   验收标准 2：Given 任务已发布成功，When 冒险家打开该任务详情，Then 页面可看到关联 Issue 的标题和任务完成标准。  
   验收标准 3：Given 我缺少任一必填项，When 我尝试发布任务，Then 系统阻止发布，并列出缺失字段。  

3. 作为委托人，我希望控制每个任务同一时间只由一名冒险家接取，以便避免多人重复劳动。  
   验收标准 1：Given 某任务处于 `待接取` 状态，When 第一名冒险家成功接取，Then 任务页面显示唯一接取人，且状态变为 `进行中`。  
   验收标准 2：Given 某任务已被接取但后来被标记为 `已放弃`，When 我将该任务重新开放，Then 任务状态变回 `待接取`，且原接取记录仍可查看。  
   验收标准 3：Given 第二名冒险家尝试接取已被占用的任务，When 我查看该任务历史，Then 页面显示本次接取被拒绝，且当前接取人保持不变。  

4. 作为委托人，我希望审核冒险家的提交结果并给出处理结论，以便推动任务按规则流转。  
   验收标准 1：Given 某任务处于 `待审核`，且提交结果满足完成标准，When 我选择 `通过`，Then 任务状态变为 `已完成`，并保存本次审核意见。  
   验收标准 2：Given 某任务处于 `待审核`，但提交结果未满足完成标准，When 我选择 `请求修改`，Then 任务状态变为 `已退回`，并保存逐项修改意见。  
   验收标准 3：Given 提交结果未关联到该任务，或缺少任务要求的成果内容，When 我尝试选择 `通过`，Then 系统阻止通过操作，且任务状态保持 `待审核`。  

5. 作为委托人，我希望查看项目内任务与 PR 的整体进度，以便掌握项目迭代状态。  
   验收标准 1：Given 某项目下存在多个任务，When 我打开项目进度页面，Then 页面显示 `待接取`、`进行中`、`待审核` 和 `已完成` 的任务数量。  
   验收标准 2：Given 某任务已经关联 PR，When 我打开该任务的进度详情，Then 页面同时显示任务状态、最新 PR 状态和最近一次审核结论。  
   验收标准 3：Given 某任务关联仓库的同步失败，When 我打开项目进度页面，Then 该任务显示 `同步异常`，且不会被计入 `已完成` 数量。  

## 管理员
1. 作为管理员，我希望审核新发布任务的清晰度和合规性，以便保证任务大厅中的内容可理解、可演示。  
   验收标准 1：Given 某新任务包含标题、技术栈、预计工作量、完成标准、奖励和关联 Issue，When 我完成审核，Then 该任务进入可见状态。  
   验收标准 2：Given 某新任务缺少完成标准，或描述与关联 Issue 无关，When 我完成审核，Then 该任务被退回修改，且不会出现在任务大厅中。  
   验收标准 3：Given 委托人已补全被退回任务的缺失信息，When 我再次审核，Then 该任务可以进入可见状态。  

2. 作为管理员，我希望维护任务分类、标签和难度选项，以便让平台中的任务展示口径保持一致。  
   验收标准 1：Given 我新增或修改了某个任务标签、分类或难度选项，When 委托人打开任务创建页面，Then 页面显示最新选项。  
   验收标准 2：Given 某标签已被多个已发布任务使用，When 我查看该标签详情，Then 页面列出所有受影响的任务。  
   验收标准 3：Given 我尝试删除仍被已发布任务使用的标签，When 我确认删除，Then 系统拒绝删除，且已发布任务的标签保持不变。  

3. 作为管理员，我希望处理仓库导入或同步异常，以便保证课程展示中的关键项目数据处于可用状态。  
   验收标准 1：Given 某仓库导入或同步失败，When 我打开异常列表，Then 页面显示项目名称、异常类型和当前状态。  
   验收标准 2：Given 某失败仓库的来源已恢复可访问，When 我发起重试，Then 该仓库状态更新为 `导入成功` 或 `同步成功`。  
   验收标准 3：Given 外部平台接口仍不可用，When 我发起重试，Then 异常状态保持 `未解决`，且平台保留该项目上一次成功同步的数据。  

4. 作为管理员，我希望处理任务、仓库和 PR 之间的关联异常，以便保证任务流转关系可追踪。  
   验收标准 1：Given 某任务、仓库和 PR 关联正确，When 我查看关系详情，Then 页面显示同一任务对应的仓库、Issue 和 PR 记录。  
   验收标准 2：Given 某提交引用了其他仓库中的 PR，When 我处理该异常，Then 任务状态保持不变，且该提交被标记为 `关联失败`。  
   验收标准 3：Given 委托人已修正错误关联并重新提交，When 我刷新该异常记录，Then 该异常被清除，任务恢复到正常审核流程。  

5. 作为管理员，我希望处理权限不足和违规操作，以便维持平台秩序并防止错误状态流转。  
   验收标准 1：Given 当前用户不是该任务的委托人且不是管理员，When 当前用户打开任务详情页，Then 页面不显示“通过”“驳回”“请求修改”等审核操作入口。
   验收标准 2：Given 委托人尝试修改不属于自己管理范围的任务，When 该操作被检查，Then 原任务内容保持不变。  
   验收标准 3：Given 某已发布任务包含与项目无关或违规内容，When 我将其下架，Then 该任务不再出现在任务大厅中。
