## 二、核心问题

### A. 背景与真实场景（约 5 分钟）

**Q1. 你目前的学习/开发状态是什么？最近三个月主要做哪些项目类型？**
追问：课程作业、个人项目、实习/工作分别占比大概多少？
记录要点：**经验阶段**、**协作经验**、常用技术栈。

**Q2. 回忆一次你“想参与但没有参与”或“中途放弃”的开源/协作经历，发生了什么？**
追问：卡在任务选择、环境搭建、理解代码、提交流程中的哪一步？（若没有开源经历，也可以回忆一次课程大作业、小组协作、实习任务）
记录要点：触发放弃的关键节点和原因。

### B. 任务发现与接取决策（约 5 分钟）

**Q3. 你在任务列表里决定“接不接”的前三重要的信息是什么？什么样的系统推荐值得你的信任？**
候选提示：任务难度、技术栈、预计时长、是否新手友好、说明完整度、审核标准清晰度；按**技
术栈**匹配、按**历史行为**、按预计完成**时长**、按**学习路径**分层，哪个更可信？
记录要点：Top3 + 理由；推荐规则偏好与信任条件。

### C. 执行支持与完成标准（约 8 分钟）

**Q4. 接到任务后，你接取后最怕什么不清楚？你最需要平台提供哪三类支持？**
候选提示：完成标准模板、贡献流程指引、参考示例、沟通渠道、导师答疑、学习资料。
记录要点：Top3 支持项 + 期望形式。

**Q5. 你认为什么样的“完成标准”才足够清晰？**
追问：你希望看到哪些可验证条目（功能点、测试要求、代码规范、文档要求）？
记录要点：完成标准必填字段建议。

### D. 审核反馈与通知机制（约 8 分钟）

**Q6. 提交后你最希望得到什么反馈？你可接受的反馈时限是多久？**
选项提示：24 小时内 / 72 小时内 / 1 周内 / 只要有响应即可。
追问：只要通过/不通过，还是要具体到可执行修改建议？是否需要学习路径建议？
记录要点：反馈深度偏好 + 反馈响应期望值。

**Q7. 任务状态变化时，哪些事件必须通知你？你更偏好什么频率和渠道？**
追问：站内、邮件、两者都要？是否需要“每日汇总”避免打扰？
记录要点：事件-渠道-频率矩阵。

### E. 激励与游戏化边界（约 8 分钟）

**Q8. 哪些因素最能让你持续参与？请从以下中选 3 个。**
候选提示：技能提升、可展示经历、维护者认可、XP/等级、徽章、排行榜、奖金。
记录要点：持续参与驱动 Top3。

**Q9. 你对“冒险家公会”的游戏化设计和“连续贡献机制”的接受边界是什么？**
追问：哪些设计会增加动力？哪些会造成压力或反感（如中断清零）？
记录要点：可接受/不可接受边界。
