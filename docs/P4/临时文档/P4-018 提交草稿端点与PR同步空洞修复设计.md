# P4-018 提交草稿端点与 PR 同步空洞修复设计

> 来源：设计评审（2026-05-31），grill-with-docs 设计访谈结论。
> 负责人：B（后端）。本文档固化"PR 同步空洞"的根因与修复方案，供 018 作者（D）、
> 前端 C 对齐。实现前先冻结此设计，实现后回填"实际改动"小节。

---

## 1. 背景：PR 同步空洞

### 1.1 现象

`SubmissionServiceImpl.createSubmission` 要求传入的 `pullRequestId` **必须已存在于本地
`pull_requests` 表**：

```java
CodePullRequest pullRequest = pullRequestRepository.findById(request.getPullRequestId())
    .orElseThrow(() -> new BusinessException("PR_NOT_FOUND", ...));
```

但**全工程没有任何代码向 `pull_requests` 表写入数据**（无 `new CodePullRequest(...)`、
无 `save`）。结果：无论前端传什么 `pullRequestId`，提交必然 `PR_NOT_FOUND`，
**整条提交链当前不可用**。

### 1.2 根因（责任链，按证据还原）

| 环节 | 事实 | 定性 |
| --- | --- | --- |
| **看板拆分（P4-002）** | 契约第 101 行的"仓库同步 / `PULL_REQUESTS` scope"是唯一写 `pull_requests` 的入口，但 39 个任务中无任何一项对应"PR 落库"。生产端从未被切分。 | **空洞根源** |
| **P4-016（适配层）** | 完成标准是"PR 通过统一适配层**访问**"（只读）。交付了 `CodePullRequest` 实体（空表映射），并在《P4-016 已知潜在问题清单》**问题 5** 明确预警"目前无 upsert 逻辑……实现 PR 同步服务时使用查找-或-更新模式"，但未指定 owner。 | 交付空表 + 预警缺口，按范围未实现 |
| **P4-018（提交后端）** | 依赖声明为 `P4-012、P4-016`，假定 016 会把 PR 备好。误把"adapter 能访问 PR"当成"本地库已有 PR"，在空表上构建提交流程。 | 依赖误读，引爆点 |

**一句话结论**：空洞由看板拆分阶段遗漏"PR 落库"任务造成；016 交付空表并在问题清单第 5 条
预警；018 误把"可访问"当"已落库"，最终引爆。

**本次修复 = 兑现 016 问题清单第 5 条承诺的"未来的 PR 同步服务"**，落在 `codehost` 域。

## 2. 设计决策

### 决策 1 — 不改提交接口，把同步补在它前面

`POST /api/v1/submissions` 入参（`pullRequestId`）由契约第 144 行定死，前端 C 正照此搭建。
改它 = 违背 P4-005 + 搞崩并行开发的前端。

- **选定**：`SubmissionController` / `CreateSubmissionRequest` / 018 测试 **零改动**。
- **否决**：让提交接口收 Gitea `prNumber` 并内部同步 —— 违背契约、越界改他人代码。

### 决策 2 — 懒同步入口 = `GET /quests/{questId}/submission-draft`

前端打开提交柜台调此接口，后端内部完成"拉取 + 落库 + 返回候选 PR"。前端开页即拿到带本地
`pullRequestId` 的 PR 下拉列表，选一个直接提交 —— **对前端是一步**。

- **选定**：懒同步塞进 `submission-draft`（契约第 143 行）。
- **否决**：实现契约第 101 行 `POST /repositories/{id}/sync` —— 需新建整个
  `RepositoryController` + syncJob 语义，体量大且前端需"先 sync 再 submit"反而不是一步；
  该重型仓库同步属 P4-017 工作台地盘，留给后续。

> **反直觉点**（未来读代码者注意）：PR 同步逻辑位于"提交草稿"接口内，而非独立的
> sync 端点。原因是 MVP 阶段以最小改动换取"前端一步提交"，避免新建仓库同步子系统。

### 决策 3 — Gitea `owner/repo` 从 `CodeRepository.sourceUrl` 解析

- **选定**：解析 `sourceUrl`（如 `http://host:3000/zhang/demo-repo` → `owner=zhang`,
  `repo=demo-repo`）。剥 `.git` 后缀、取末两段路径，带兜底。`sourceUrl` 为 `nullable=false`
  必填字段，导入即有值，不依赖绑定关系。
- **否决**：用 `CodeHostAccountBinding.externalUsername` 取 owner —— 引入绑定依赖与额外失败
  模式（owner 未绑定 Gitea 怎么办）。

### 决策 4 — `submission-draft` 列出仓库**全部** PR，不按作者过滤

- **选定（甲）**：`GET /repos/{owner}/{repo}/pulls?state=all`，全量 upsert，返回给前端。
  与决策 3（不碰 binding）、与 018 现状（只校验"PR 属于该任务仓库"、不校验作者）口径一致，
  不引入新不变量。
- **否决（乙）**：按冒险家 Gitea 用户名过滤 —— 把 binding 依赖请回来，且与 018 现状不一致。
- **延期增强**：见 §5。

## 3. 接口与数据契约

### 3.1 端点

```
GET /api/v1/quests/{questId}/submission-draft
```

返回字段（契约第 143 行）：

| 字段 | 来源 |
| --- | --- |
| `questId` | `Quest.questId` |
| `repository` | `Quest.repository`（id / name / sourceUrl 摘要） |
| `branch` | **数据层无来源**（`QuestAssignment` 无 branch 字段）→ 暂用 `repository.defaultBranch` 兜底，见 §5。**注：`CodeRepository` 当前无 `getDefaultBranch()` getter，需补** |
| `pullRequests[]` | 同步后的本地 PR（`pullRequestId` / `externalPrId` / `title` / `status` / `sourceBranch` / `targetBranch` / `externalUrl`） |
| `completionCriteria` | `Quest.completionCriteria` |

错误码：`QUEST_NOT_FOUND`、`ASSIGNMENT_NOT_FOUND`（仅 ACTIVE assignee 可取草稿）。

### 3.2 Gitea PR 状态 → 本地 `status` 映射（契约第 40 行）

| Gitea `state` + `merged` | 本地 `status` |
| --- | --- |
| `open` | `OPEN` |
| `closed` + `merged=true` | `MERGED` |
| `closed` + `merged=false` | `CLOSED` |

> 契约第 40 行的 `CHECKING`/`REVIEW_REQUESTED`/`CHANGES_REQUESTED` 为工作台展示扩展，
> 同步阶段只产出 `OPEN`/`MERGED`/`CLOSED` 三态。

### 3.3 upsert（兑现 016 问题 5）

唯一键 `uk_pr_repository_external (repository_id, external_pr_id)`：

```java
CodePullRequest pr = repo.findByRepositoryRepositoryIdAndExternalPrId(repoId, externalId)
    .orElseGet(() -> new CodePullRequest(repository, externalId, title, srcBranch, tgtBranch, status, url));
pr.setStatus(latestStatus);
// 注：mergedAt 无来源（PrInfo 不携带 merged_at），isMerged() 只依赖 status，MVP 跳过 setMergedAt
repo.save(pr);
```

## 4. 改动清单（全在 `codehost` 域 + 一个 draft 端点）

| 文件 | 改动 |
| --- | --- |
| `GiteaAdapter` / `GiteaAdapterImpl` | 新增 `List<PrInfo> listPulls(owner, repo)` → `GET /repos/{owner}/{repo}/pulls?state=all`；并捕获 `HttpClientErrorException` 转 `BusinessException`（问题 C） |
| `dto/PrInfo` | **补两个字段**：`baseBranch`（目标分支，因 `CodePullRequest.target_branch` 非空，从 `base.ref` 读）+ `htmlUrl`（PR 链接，作 `CodePullRequest.externalUrl`，从 `html_url` 读）。原 `getPullRequest`/`toPrInfo` 同步适配 |
| `CodeRepository` | **补 `getDefaultBranch()` getter**（字段已有，getter 缺失，Bug 1） |
| `CodePullRequestSyncService`（新） | `sourceUrl` 解析 owner/repo + 状态映射 + upsert |
| `SubmissionDraftController` + `SubmissionDraftService`（新，挂 `/quests/{questId}/submission-draft`） | 校验 ACTIVE assignee → 调同步 → 组装返回 |

**不动**：`SubmissionController`、`CreateSubmissionRequest`、`SubmissionServiceImpl`、`ReviewServiceImpl` 及其全部测试。

## 5. 延期项 / 新增已知问题

| 编号 | 描述 | 处理时机 |
| --- | --- | --- |
| **A** | **不校验 PR 作者归属**（决策 4 甲）。理论上冒险家可选他人 PR。"只能提交自己 PR"需引入 binding 比对，留作后续增强。 | 安全加固阶段 |
| **B** | **`submission-draft.branch` 无数据来源**（`QuestAssignment` 无 branch 字段）。暂用 `repository.defaultBranch` 兜底。若需真实工作分支，需在 Assignment 上加字段或从选中 PR 的 `sourceBranch` 推导。 | 联调（P4-029）前 |
| **C** | **顺带修 016 问题 1**：`GiteaAdapterImpl` 不捕获 `HttpClientErrorException`，Gitea 4xx 透传为 500。本次同步链会触发它（仓库不存在 / token 失效），一并转 `BusinessException`（`CODE_HOST_UNAVAILABLE` 等）。 | 本次实现已处理（`execute()` 包装，404→`CODE_HOST_RESOURCE_NOT_FOUND`，其余→`CODE_HOST_UNAVAILABLE`） |
