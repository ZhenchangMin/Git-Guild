## 2. 设计决策

### 决策 1 — 只做 quest 推荐端点

实现 `GET /api/v1/recommendations/quests`（契约 §74）。另外两个端点
（`/quests/{questId}/contributors`、`/reasons`）不在 P4-025 看板范围内，延后。

### 决策 2 — 真实规则推荐（非 Mock）

按 Adventurer 历史 + Quest 属性计算匹配分，而非演示假数据。

### 决策 3 — 技术偏好从已完成 Quest 反推

Adventurer（`BEGINNER` 角色）没有自定义 techStack 字段，
从其 `ContributionRecord` 关联的已完成 Quest 的 `techStackJson` 解析反推。

### 决策 4 — 权重 (a) 技术导向：w1=0.6, w2=0.4

综合分 `score = 0.6 × techMatch + 0.4 × difficultyMatch + beginnerBoost`。
新 Adventurer（无历史）→ techMatch=0 → 退化为纯 difficulty 排序，冷启动友好。

## 3. 推荐算法

### 3.1 候选集

```
候选 = { q | q.status ∈ {PUBLISHED, IN_PROGRESS} }
     − { q | ∃ ACTIVE Assignment(q, adventurer) }
     − { q | ∃ ContributionRecord(q, adventurer) }
```

### 3.2 Adventurer 画像

| 画像维度 | 来源 | 计算 |
| --- | --- | --- |
| 技术栈偏好 | 已完成 Quest 的 `techStackJson`（如 `["Java","Spring"]`） | 解析 JSON，统计频率 |
| 难度舒适区 | 已完成 Quest 的 `difficulty` | Difficulty 数字映射：A=4, B=3, C=2, D=1 → 取均值 |

### 3.3 候选打分

**Tech 匹配度（0–1，权重 0.6）：**
```
techMatch = |adventurerTechs ∩ questTechs| / |questTechs|
```
questTechs 为空时 → techMatch = 0（退化为纯 difficulty 排序）。

**Difficulty 匹配度（0–1，权重 0.4）：**
```
difficultyMatch = 1 − |questDiff − adventurerAvgDiff| / 3
```
随偏离舒适区越远平滑下降。

**新手友好 boost：** `beginnerFriendly=true` 时，若 Adventurer ≤ Lv2 且 Quest 难度 D → +0.2。

**综合分：**
```
score = 0.6 × techMatch + 0.4 × difficultyMatch + beginnerBoost
```

### 3.4 StrongMatch 与 Reasons

- `strongMatch = true` 当 `score ≥ 0.7`。
- `reasons[]` 生成规则：techMatch ≥ 0.5 → "技术栈匹配"；difficultyMatch ≥ 0.7 → "难度适合你"；beginnerFriendly 激活 → "新手友好"；无任何命中 → "推荐尝试"。

### 3.5 分页与排序

候选按 score 降序排列，取 top-N（默认 5，最大 20）。不涉及复杂分页。

## 4. 改动清单

**新增（`recommendation` 域，自有地盘）：**

| 文件 | 职责 |
| --- | --- |
| `recommendation/service/RecommendationService(+Impl)` | 核心：画像构建 + 候选过滤 + 打分 + 排序 + reasons |

**补 repository 方法：**
- `ContributionRecordRepository`：补 `findByUserUserId(Long userId)` → 查 Adventurer 已完成 quest 列表
- `QuestAssignmentRepository`：补 `findByAssigneeUserIdAndStatus(Long userId, AssignmentStatus status)` → 查 ACTIVE 接取

**不改动的边界：** Quest 状态机、Submission/Review 流程、Growth 发放逻辑均零改动。
