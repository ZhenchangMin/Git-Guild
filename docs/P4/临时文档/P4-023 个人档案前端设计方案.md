# P4-023 个人档案前端设计方案

> 本文档描述 Git-Guild 个人档案页的前端设计方案，覆盖布局结构、各区块内容、数据对接、视觉风格和交互行为。基于 P4-022 成长激励后端设计，整合 `GET /api/v1/users/me/growth` 接口已有的返回数据，并扩展趣味数据展示。

---

## 1. 页面定位与入口

个人档案是冒险者在 Git-Guild 中的"身份名片"，聚合展示用户身份、成长数据、能力画像和贡献历程。

- **入口**：公会大厅右侧"成长档案"按钮 → `/profile`（已有路由配置）
- **返回**：左上角返回按钮 → 公会大厅 `/hall`
- **路由**：`name: 'profile'`，已有 `meta: { requiresAuth: true, roles: ['ADVENTURER', 'MAINTAINER', 'ADMIN'] }`

---

## 2. 页面整体布局

页面采用**单列纵向滚动**结构，复用现有的 `scene` + `profile-route-mode` 背景层。内容从上到下分为 **6 个区块**：

```
┌─────────────────────────────────────────┐
│  ① 冒险者执照（身份主卡）                │
│     头像/昵称 + 等级/XP进度条/称号/数据  │
├─────────────────────────────────────────┤
│  ② 统计指标卡片（词云风格）              │
│     数值越大→面积越大、字号越大          │
├─────────────────────────────────────────┤
│  ③ 趣味数据双栏（2 张图表卡）            │
│     左：难度攀升曲线  右：技能雷达图      │
├─────────────────────────────────────────┤
│  ④ 技能标签区                            │
│     技术栈标签列表                        │
├─────────────────────────────────────────┤
│  ⑤ Tab 切换内容区                        │
│     [贡献历程] [成就徽章] [成长里程碑]    │
├─────────────────────────────────────────┤
│  ⑥ 快捷操作栏                            │
│     浏览任务 | 进入工作台 | 排行榜        │
└─────────────────────────────────────────┘
```

## 3. 各区块详细设计

### 3.1 冒险者执照（身份主卡）

#### 设计原则

- **统一冒险家身份**：所有用户（包括委托人）在个人档案中均以冒险家身份展示，不区分角色。委托人只是比普通冒险家多了一个"发布委托"的能力，但在档案层面大家是平等的公会成员。委托人的管理职责数据（已发布任务数、已审核数等）放在工作台展示，档案页只展示冒险家视角的成长数据。
- **可编辑**：执照卡包含"编辑资料"入口，支持修改头像和座右铭。

#### 布局结构

**容器**：一张完整的 `glass-ledger` 卡片，顶部带"冒险者执照"标题栏（深色半透明背景 + 金色小字），右侧显示执照编号（用户 ID 补零，如 `No. 00005`）。

### 3.2 统计指标卡片（词云风格）

**设计理念**：仿照词云图的视觉逻辑——数值越大的指标，卡片面积越大、数字字号越大，形成视觉层次。但设置面积下限，保证所有卡片内容可完整阅读。

**布局**：4 张卡片放在同一个 `flex` 容器中，`flex-wrap: wrap`，每张卡片的 `flex-grow` 按数值比例动态分配。

**动态面积算法（百分位 + 重要性权重）**：

各指标数值量级差异大（如 XP 720 vs 待审核 1），直接用原始数值做线性映射会让大数值指标碾压其余，因此引入重要性权重进行调节。

1. 每个指标定义一个重要性权重 `weight`（待定，见下方权重表）
2. 计算加权值 `weightedVal = value × weight`
3. 取所有指标加权值中的最大值 `maxWeighted`
4. 每张卡片的 `flex-grow` = `Math.max(0.6, weightedVal / maxWeighted)`，其中 `0.6` 为最小比例系数
5. 卡片设置 `min-width`（如 120px）和 `min-height`（如 100px），确保内容不溢出

**重要性权重表（待定）**：

| 指标 | weight | 说明 |
|------|--------|------|
| 已完成任务 | `TBD` | 核心成长指标 |
| 累计 XP | `TBD` | 数值大但不应视觉碾压 |
| 贡献仓库数 | `TBD` | 反涉广度 |
| 待审核 | `TBD` | 反映当前活跃度 |

> 权重需要在接入真实数据后，根据实际数值分布和视觉效果进行调参。

**动态字号算法**：
- 数字字号在 `[1.4rem, 2.8rem]` 区间内，按加权值比例映射
- 字号 = `1.4 + (weightedVal / maxWeighted) * 1.4` rem
- 加权值最大的卡片字号 2.8rem，最小的卡片字号 1.4rem

**单张卡片内容**（从上到下，居中排列）：
- 小图标（SVG，低透明度金色，描边风格）
- 大数字（衬线体，亮金色，字号动态）
- 小字标签（低透明度，字号固定）

**四个指标**：

| 图标 | 标签 | 数据来源 |
|------|------|---------|
| 卷轴 | 已完成任务 | `growth.completedQuestCount` |
| 星形 | 累计 XP | `growth.totalXp` |
| 方格 | 贡献仓库数 | 前端聚合（去重统计 `contribution_records` 中不同仓库数量） |
| 时钟 | 待审核 | 前端统计状态为 `PENDING_REVIEW` 的提交数 |

**交互**：hover 时边框亮度提升，带微弱金色光晕。

**示例效果**（假设数据：已完成 6、累计 XP 720、贡献仓库 3、待审核 1；假设权重：已完成 1.0、XP 0.5、仓库 0.8、待审核 0.9）：
- 加权值：已完成 = 6×1.0 = 6.0，XP = 720×0.5 = 360，仓库 = 3×0.8 = 2.4，待审核 = 1×0.9 = 0.9
- maxWeighted = 360
- 累计 XP → flex-grow = 1.0，字号 2.8rem（面积最大）
- 已完成任务 → flex-grow ≈ 0.6（触发下限），字号 ≈ 1.42rem
- 贡献仓库 → flex-grow = 0.6（触发下限），字号 ≈ 1.41rem
- 待审核 → flex-grow = 0.6（触发下限），字号 ≈ 1.4rem

> 以上权重为示例，实际权重待接入真实数据后调参确定。

### 3.3 趣味数据双栏

**布局**：两列等宽，各一张 `glass-ledger` 卡片，内含标题和图表。

#### 左栏 — 难度攀升曲线（可交互）

一个可拖动时间轴的 SVG 折线图，展示冒险者接取任务难度随时间的变化趋势。

**图表元素：**
- **X 轴**：任务完成日期（如 05-10、05-15、05-20...）
- **Y 轴**：难度等级，4 级制（A 最难）：
  - D = 入门
  - C = 初级
  - B = 中级
  - A = 高级
- **折线**：金色渐变描边（`#c8851a` → `#f1b756` → `#ffe0a3`），圆角连接
- **面积填充**：折线下方半透明金色渐变，从上到下透明度递减
- **数据点**：每个节点有外圈光晕动画（脉冲呼吸效果）+ 实心金色圆点
- **网格线**：4 条水平虚线对应 4 个难度等级（D/C/B/A）
- **底部文字**："当前挑战水平：B"（取最后一个数据点的难度标签）

**时间轴缩放交互：**

图表下方有一个双滑块时间轴控件，支持用户分别拖动上下界来调整可视范围：

```
┌─────────────────────────────────────────┐
│            折线图主区域                   │
│     （只显示当前可视范围内的数据点）       │
├─────────────────────────────────────────┤
│  ┌───────────────────────────────────┐  │
│  │░░░░░░░░░████████████░░░░░░░░░░░░░│  │  ← 时间轴轨道
│  │       ◀━━━━━━━━━━━━━━▶           │  │
│  │      下界滑块      上界滑块        │  │  ← 两个可拖动滑块
│  └───────────────────────────────────┘  │
│  04-01    05-01    05-15    05-30       │
└─────────────────────────────────────────┘
```

- **时间轴轨道**：底部显示完整时间范围的横向轨道，可见数据点分布密度
- **下界滑块（左）**：可左右拖动，设定可视范围的起始日期
- **上界滑块（右）**：可左右拖动，设定可视范围的结束日期
- **两个滑块之间的区域**：高亮显示，标示当前折线图的时间窗口
- **默认状态**：下界在最左、上界在最右，显示完整时间线

**缩放行为：**
- **拖动下界向右**：起始日期后移 → X 轴变密集，数据点间距拉开
- **拖动上界向左**：结束日期前移 → 同上
- **拖动下界向左 / 上界向右**：范围扩大 → X 轴变稀疏，概览更多趋势
- **两滑块不能交叉**：下界最大值受限于上界当前位置，反之亦然
- **数据点不足时**：如果可视范围内只有 1 个数据点，居中显示，不绘制连线

**其他交互：**
- hover 数据点时显示浮动气泡，包含：任务名称、完成日期、难度标签、获得 XP
- 折线图和时间轴之间的过渡带平滑动画

#### 右栏 — 技能雷达图（可编辑标签）

一个可自定义标签的 SVG 正多边形雷达图，展示冒险者在不同技术维度的能力分布。

**数据池**：系统追踪的技能标签总数多于雷达图可显示的数量（如追踪 12-15 个技能），用户从中选择一部分展示在雷达图上。

**标签选择器**：雷达图下方或右侧提供标签选择区，展示所有可用技能标签：
- 每个标签为可点击的胶囊按钮，已选中态有金色高亮边框，未选中态为低透明度
- 点击标签切换选中/取消选中
- 雷达图实时更新：选中的标签作为雷达轴，取消后该轴消失，多边形自动重绘
- 选中数量建议范围：3-8 个（少于 3 个雷达图无意义，多于 8 个会拥挤，界面给出提示）

**雷达图元素：**
- **轴数**：等于当前选中的标签数（3-8 个），动态变化
- **4 层同心圈**：作为刻度参考线（25% / 50% / 75% / 100%）
- **轴线**：从中心到顶点的低透明度直线
- **数据区域**：半透明金色填充 + 金色描边，圆角连接
- **数据点**：每个轴顶点有实心圆点（亮金色 + 深色描边）
- **轴标签**：显示在多边形外部，根据位置自动调整文本对齐（左/中/右）
- **底部摘要**：
  - "最强：前端组件 85%"（金色高亮）
  - "待提升：后端集成 40%"（普通灰色）

**重绘动画**：标签切换时，多边形平滑过渡到新形状（顶点坐标插值动画），而非瞬间重绘。

**技能数据池示例**（mock，0-100 数值）：

| 技能 | 默认选中 | 数值来源 |
|------|---------|---------|
| 前端组件 | ✓ | 贡献记录中涉及 Vue 组件的任务 |
| 状态管理 | ✓ | 涉及 Pinia/Vuex 的任务 |
| CSS 样式 | ✓ | 涉及样式/布局的任务 |
| 后端集成 | ✓ | 涉及后端 API/适配器的任务 |
| 文档撰写 | ✓ | 涉及文档/README 的任务 |
| 异常处理 | ✓ | 涉及错误状态/异常的任务 |
| 表单校验 | | 涉及表单/校验的任务 |
| Gitea 集成 | | 涉及 Gitea API 的任务 |
| 测试编写 | | 涉及单元测试/集成测试的任务 |
| 性能优化 | | 涉及缓存/懒加载的任务 |
| 交互设计 | | 涉及 UX/动效的任务 |

**数据来源**：目前使用 mock 数据，后续从贡献记录的 `skills` 字段聚合，按任务关联的技能标签自动统计每个维度的数值。

### 3.4 技能标签区

**容器**：一张 `glass-ledger` 卡片，标题"技能图谱"，副标题"冒险中积累的技术栈"。

- 标签横向自动换行排列（`flex-wrap`）
- 每个标签为圆角胶囊样式，标签右侧小数字表示出现次数
- 出现次数越多，标签边框品质越高，仿照游戏装备品质分级：

**三阶品质边框：**

| 品质 | 出现次数 | 边框样式 | 背景 | 文字颜色 |

### 3.5 Tab 切换内容区

**Tab 栏**：三个按钮横向排列在深色半透明容器中。

每个 Tab 按钮包含：SVG 图标（描边风格）+ 文字。选中态有金色边框和微弱背景色。

#### Tab 1：贡献历程（默认选中）

**布局**：垂直时间线。

- **左侧**：圆点节点 + 竖线连接，最底部节点没有连线。圆点为金色描边 + 深色填充，带微弱光晕。
- **右侧**：卡片列表，每张卡片包含：
  - 顶行：完成日期（灰色小字）+ 获得 XP（金色高亮，如 `+180 XP`）
  - 任务标题（标题字体，亮金色）
  - 仓库名 + PR 编号（小字灰色，如 `git-guild / frontend · PR #12`）
  - 一句话贡献摘要（普通灰色正文）
  - 技术标签行（mini 圆角标签，最低透明度金色边框）

**交互**：卡片 hover 时边框亮度提升。

#### Tab 2：成就徽章

**布局**：三列网格（`grid-template-columns: repeat(3, 1fr)`）。

**已达成徽章**：
- SVG 星形徽章图标（金色，5 角星 + 圆形底座）
- 徽章名称（亮金色标题字体）
- 达成描述（灰色正文）
- 达成日期（最低透明度小字）
- 卡片右上角"佩戴"按钮（已佩戴时显示金色勾标记，未佩戴时 hover 显示"佩戴"文字）
- 整体卡片 hover 时轻微上浮 + 光晕

**佩戴展示**：
- 用户可从已达成徽章中选择一个佩戴，佩戴后该徽章图标显示在冒险者执照卡的身份区（座右铭下方，以小图标 + 徽章名称的形式展示）
- 同一时间只能佩戴一个徽章，点击其他徽章的"佩戴"按钮会自动替换
- 未佩戴任何徽章时，执照卡该区域不显示
- 后端需新增 `display_badge_id` 字段（可空），前端暂可用 mock 管理状态

**未达成徽章**：
- SVG 锁形图标（虚线圆底座 + 锁扣），灰色
- 徽章名称 + 达成条件
- 整体半透明（`opacity: 0.4`）+ 灰度滤镜（`filter: grayscale(0.6)`）
- 不显示"佩戴"按钮

**预留徽章类型**：

| 徽章 ID | 名称 | 达成条件 |
|---------|------|---------|

#### Tab 3：成长里程碑

**设计理念**：不是 XP 流水的逐条罗列，而是提炼成长过程中的关键节点——只有真正有意义的事件才构成里程碑。

**布局**：垂直时间线（复用 Tab 1 的时间线样式），但节点更大、更突出。

**里程碑节点类型**：

| 类型 | 图标 | 触发条件 | 展示内容 |
|------|------|---------|---------|
| 首次冒险 | 启程卷轴 | 完成第一个任务 | 任务名称、完成日期 |
| 等级提升 | 上升箭头 | 等级发生变化 | 新等级、累计 XP |
| 徽章达成 | 星形徽章 | 获得任意徽章 | 徽章名称、达成描述 |
| 连续突破 | 火焰图标 | 连续完成 N 个任务 | 连续次数 |
| 技术拓展 | 分支图标 | 首次在新仓库贡献 | 仓库名称 |

**单个里程碑卡片**：
- 左侧：类型图标（大号，金色描边），带时间轴竖线连接
- 右侧：事件标题（如"等级提升 — Lv 3 → Lv 4"）+ 日期 + 一句话描述
- 当前里程碑（最新事件）卡片边框高亮，带微弱金色光晕

**空态**：暂无里程碑时显示引导文案："完成你的第一个任务，开启冒险旅程。"

**数据来源**：从贡献记录、XP 流水、徽章状态中聚合提炼关键节点，目前使用 mock 数据。

### 7.2 本次需新增的接口

| 接口 | 方法 | 说明 | 对应区块 | 改动范围 |
|------|------|------|---------|---------|
| `/api/v1/users/me/avatar` | POST | 上传头像图片，返回头像 URL。支持常见图片格式（jpg/png/gif），建议限制文件大小（如 ≤ 2MB） | ① 执照卡 | User 实体新增 `avatar_url` 字段（VARCHAR(512)，可空）；新增 Controller 方法 |
| `/api/v1/users/me` | PATCH | 更新用户资料，请求体包含 `motto`（VARCHAR(200)，可空）等可编辑字段 | ① 执照卡（编辑座右铭） | User 实体新增 `motto` 字段；新增或扩展现有 Controller |

| 接口 | 方法 | 说明 | 对应区块 | 备注 |
|------|------|------|---------|------|
| `/api/v1/users/me/contributions` | GET | 贡献记录列表，分页。返回：任务标题、仓库名、PR 编号、完成日期、获得 XP、贡献摘要、技能标签数组、难度等级 | ② 统计卡片（贡献仓库数、待审核）、③ 难度曲线、④ 技能标签、⑤ 贡献历程、⑤ 成长里程碑 | P4-022 标注待 P4-023 需要时补充。**多个区块共用此接口**，是最核心的延后依赖 |

### 7.4 前端聚合逻辑（无需独立接口）

以下数据不依赖新接口，前端从已有数据中计算：

| 数据 | 计算方式 | 对应区块 |
|------|---------|---------|
| 公会称号 | 根据 `growth.level` 查称号梯度表 | ① 执照卡 |
| Code 龄 | `User.createdAt` 与当前日期的相对天数 | ① 执照卡 |
| 贡献仓库数 | 去重统计 `contributions` 中不同仓库数量 | ② 统计卡片 |
| 待审核数 | 统计 `contributions` 中状态为 `PENDING_REVIEW` 的条数 | ② 统计卡片 |
| 难度曲线 | 从 `contributions` 提取 `(完成日期, difficulty)` 序列 | ③ 折线图 |
| 技能雷达 | 从 `contributions` 的 `skills` 数组聚合各维度计数 | ③ 雷达图 |
| 技能标签 | 从 `contributions` 的 `skills` 数组聚合计数 + 品质分级 | ④ 技能标签 |
| 成长里程碑 | 从 `contributions` + `xp-transactions` + 徽章状态聚合关键节点 | ⑤ 里程碑 |
