## 问题 1 — Gitea 异常透传为 HTTP 500

**文件：** `GiteaAdapterImpl.java`（所有四个方法）

**描述：**
`GiteaAdapterImpl` 的方法不捕获 `HttpClientErrorException`。当 Gitea 返回 4xx（如仓库不存在返回 404、Token 无效返回 401）时，Spring RestClient 抛出 `HttpClientErrorException`，被 `GlobalExceptionHandler` 的 catch-all 捕获后一律返回 HTTP 500 + `"服务器内部错误"`。

**影响：**
调用方（P4-018 提交后端等）无法区分"仓库不存在"和"服务器内部错误"，前端也无法展示有意义的错误提示。

**建议修复时机：** P4-018 或 P4-017 开发期间，在 `GiteaAdapterImpl` 中捕获 `HttpClientErrorException` 并转换为带业务码的 `BusinessException`（如 `GITEA_REPO_NOT_FOUND`、`GITEA_PR_NOT_FOUND`、`CODE_HOST_UNAVAILABLE`），或在 `GlobalExceptionHandler` 中添加专项处理。

## 问题 2 — GITEA_TOKEN 未配置时应用静默启动，运行时才报 401

**文件：** `GiteaProperties.java`、`application.yml`

**描述：**
`application.yml` 中 `gitea.token: ${GITEA_TOKEN:}` 在 `GITEA_TOKEN` 环境变量未设置时绑定为空字符串。`GiteaProperties` 没有 `@Validated` / `@NotBlank` 校验，Spring 上下文正常启动。所有 Gitea API 调用发送 `Authorization: token `（空 token），Gitea 返回 401，错误在运行时才暴露且上下文信息不足。

**影响：**
开发者本地漏配环境变量时，应用看似启动成功，第一次触发 Gitea 相关功能才报错，难以定位是配置问题还是代码 Bug。

**建议修复时机：** 联调准备阶段（P4-029 前）。在 `GiteaAdapterImpl` 构造方法中加一行快速失败：
```java
if (props.token() == null || props.token().isBlank()) {
    throw new IllegalStateException("gitea.token 未配置，请设置 GITEA_TOKEN 环境变量");
}
```
或在 `GiteaProperties` 上加 `@Validated` 并在 `token` 上加 `@NotBlank`。

## 问题 3 — listIssues 和 listBranches 无分页，最多返回 50 / 20 条

**文件：** `GiteaAdapterImpl.java`，`listIssues()`（第 38 行）、`listBranches()`（第 55 行）

**描述：**
- `listIssues` 硬编码 `&limit=50`，只请求第一页，Issues 超过 50 条时静默截断。
- `listBranches` 未传 `limit`，依赖 Gitea 默认值（20 条/页），同样只取第一页。
- 两个方法均无分页循环，超出的数据无任何错误或警告。

**影响：**
大型仓库（Issue 超 50 条、分支超 20 条）的同步结果不完整，但不报错。目前 P4 Demo 场景使用测试仓库，数量较少，不会触发该问题。

**建议修复时机：** 全功能同步阶段（P4-016 扩展或 P4-018 同步服务）。需要加分页循环，读取 Gitea 响应头 `X-Total-Count` 或判断返回条数少于 `limit` 时停止翻页。

## 问题 4 — CodeHostAccountBinding 缺少唯一约束注解，Hibernate update 模式不会建索引

**文件：** `CodeHostAccountBinding.java`，`@Table` 注解

**描述：**
数据库设计文档定义了 `UNIQUE KEY uk_bindings_host_account (host_type, external_account_id)`，但 `@Table` 注解未声明 `uniqueConstraints`，导致：
- `ddl-auto: update` 从零建表时不创建该唯一索引。
- 同一 Gitea 账号可被绑定到多个 Git-Guild 用户，违反业务约束，且不会报错。

**影响：**
本地从零初始化数据库时缺少该约束；若通过 `init.sql` 手动建表则不受影响（init.sql 已包含该索引）。

**建议修复时机：** 下次修改 `CodeHostAccountBinding` 实体时一并加上：
```java
@Table(
    name = "code_host_account_bindings",
    uniqueConstraints = @UniqueConstraint(
        name = "uk_bindings_host_account",
        columnNames = {"host_type", "external_account_id"}
    )
)
```

## 问题 5 — CodePullRequest 重复同步时没有 upsert 保护

**文件：** `CodePullRequestRepository.java`，未来的 PR 同步服务

**描述：**
数据库有 `UNIQUE KEY uk_pr_repository_external (repository_id, external_pr_id)`，但目前无 upsert 逻辑。若同步服务对同一个 PR 调用两次 `save(new CodePullRequest(...))`，第二次会触发 `DataIntegrityViolationException`（DB 唯一约束冲突）。

**影响：**
Gitea Webhook 重复投递（常见场景）或定时同步重跑时，未处理的异常会导致 HTTP 500。

**建议修复时机：** 实现 PR 同步服务时，使用"查找-或-更新"模式：
```java
CodePullRequest pr = repo.findByRepositoryRepositoryIdAndExternalPrId(repoId, externalId)
    .orElse(new CodePullRequest(...));
pr.setStatus(latestStatus);
repo.save(pr);
```
