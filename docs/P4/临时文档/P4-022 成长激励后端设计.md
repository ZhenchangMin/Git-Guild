# P4-022 成长激励后端设计

> 本文档固化"审核通过 → 发放成长激励"的设计，018/020 审核流作者、
> 前端: 实现前先冻结此设计，实现后回填"实现状态"小节。

---

## 1. 背景与目标

**目标（看板 P4-022 完成标准）**：审核通过后增加 XP、更新等级、生成贡献记录。

**现状探查：**
- 成长域是空地：`growth/` 包仅 `package-info.java`，`User` 实体无 xp/level 字段。
- DB 三张表已在 `docs/P4/database/init.sql` 定义好：
  | 表 | 性质 | 关键字段 |
  | --- | --- | --- |
  | `growth_profiles` | 每人一条聚合（UNIQUE `user_id`） | `total_xp`、`level`、`completed_quest_count` |
  | `xp_transactions` | 流水账（只增） | `user_id`、`quest_id`（可空）、`amount`、`reason` |
  | `contribution_records` | 贡献记录（UNIQUE `user_id`+`quest_id`） | `summary`、`completed_at` |
- `Quest.rewardXp`（`@Min(1)`，无上限）已存在；审核通过响应**早已回显** `quest.getRewardXp()`，但从未真正落库发放——P4-022 即把"回显"变成"真发"。

---

## 2. 设计决策

### 决策 1 — 发放钩子：在审核通过处直接调用，同事务

加 XP 必须由"审核通过"触发，而该逻辑位于 `ReviewServiceImpl.applyDecision` 的 APPROVED 分支
（D 写的 018/020 代码）。无法从外部旁路。

- **选定（甲）**：APPROVED 分支直接调 `growthService.grantQuestCompletion(submitter, quest)`。
  与审核**同一事务**：审核回滚则 XP 一并回滚，杜绝"审核失败但 XP 已加"。最简单、原子、易测。
- **否决（乙）**：Spring 事件解耦。仍需在 `applyDecision` 加"发事件"行（一样动 D 的代码），
  多一层间接，为单一消费者引入事件总线 = 过度设计。将来若 P4-024 通知也要监听"审核通过"，
  再一起重构为事件不迟。

> P4-022 是分配给 B 的任务且本就要挂进审核流，由 B 在此加钩子是正当的，不算越界。

### 决策 2 — XP 数额 = `quest.rewardXp`

任务自带 `rewardXp`，审核响应也一直在回显它。直接用，不引入固定值或难度系数。

### 决策 3 — 升级公式：线性可调

- **选定（甲）**：`level = totalXp / XP_PER_LEVEL + 1`，常量 `XP_PER_LEVEL = 100`。
  （0–99→Lv1，100–199→Lv2…）。`nextLevelXp = level × XP_PER_LEVEL`（契约 §196 需此字段）。
  从 `total_xp` **重算** level（而非自增），天然幂等、不怕重复。简单、可预测、对任意 `rewardXp` 量级成立。
- **否决（乙）**：递增 RPG 曲线（如升 Lv L 需累计 `50·L·(L-1)`）。有游戏感但公式复杂、阈值拍脑袋，
  MVP 不必要；想换只需改 `recordQuestCompletion` 一处。

### 决策 4 — 幂等：贡献记录唯一键兜底 + 状态机天然保证

- 审核通过会把 quest 置为 `COMPLETED`，而 `createSubmission` 拒绝非 `IN_PROGRESS/IN_REVIEW` 的 quest，
  故**首次通过即锁死后续提交，天然只发一次**。
- 兜底：`grantQuestCompletion` 发放前查 `contribution_records` 是否已有 (user, quest)，有则整体跳过，
  对应 DB `UNIQUE KEY uk_contributions_user_quest`。

## 3. 核心逻辑

`GrowthService.grantQuestCompletion(User submitter, Quest quest)`（`@Transactional`，由审核事务带入）：

```
1. if contributionRecordRepository.existsByUserUserIdAndQuestQuestId(userId, questId): return  // 幂等跳过
2. GrowthProfile profile = growthProfileRepository.findByUserUserId(userId)
       .orElseGet(() -> new GrowthProfile(submitter));        // xp=0, level=1, count=0
   profile.recordQuestCompletion(quest.getRewardXp());        // total_xp += xp; level=重算; count++
   growthProfileRepository.save(profile);
3. xpTransactionRepository.save(new XpTransaction(submitter, quest, quest.getRewardXp(), "QUEST_COMPLETED"));
4. contributionRecordRepository.save(
       new ContributionRecord(submitter, quest, quest.getRepository(), quest.getTitle(), now));
```

`GrowthProfile.recordQuestCompletion(int xp)`（富领域行为）：
```java
this.totalXp += xp;
this.level = this.totalXp / XP_PER_LEVEL + 1;   // XP_PER_LEVEL = 100
this.completedQuestCount += 1;
```
