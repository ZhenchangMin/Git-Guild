# P4-005 前后端接口契约清单

## 1. 文档目的

本文档用于 P4 编码阶段统一前端页面、后端 API、字段、状态和错误提示。后续前端接入、后端开发、Gitea 适配、联调测试均以本清单为初始契约。

本清单遵循以下已有设计：

- 平台采用自有账号体系（参见 `docs/P3/API规范/API规范-"用户登录与认证"部分.md`）。
- 代码托管底座集成自建 Gitea；GitHub 仅作为外部仓库导入与单向同步来源（参见 `docs/P3/API规范/API规范-"代码支撑平台"部分.md`）。
- 任务、接取、审核流程与 `docs/P3/API规范/API规范-"任务与悬赏"部分.md`、`docs/P3/API规范/API规范-"提交评价"部分.md`、`docs/P3/API规范/API规范-"管理员审核"部分.md` 保持一致。
- MVP 优先完成仓库导入、Issue 浏览、任务发布、任务接取、成果提交、审核、轻量成长。
- 推荐匹配、排行榜、徽章纳入 P4 范围，但优先级为 P2，不阻塞主链路。
- 数据库字段以 `docs/P4/database/init.sql` 为准。

## 2. 通用约定

| 项      | 约定                                                                                                                                                                       |
| ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| API 前缀 | `/api/v1`                                                                                                                                                                |
| 鉴权     | 登录后使用 `Authorization: Bearer <JWT_TOKEN>`                                                                                                                                |
| 成功响应   | `{ "code": "SUCCESS", "message": "OK", "data": ..., "timestamp": "...", "traceId": "..." }`                                                                            |
| 失败响应   | `{ "code": "<ERROR_CODE>", "message": "...", "details": "...", "timestamp": "...", "traceId": "..." }`                                                                  |
| 分页字段   | 列表接口请求统一 `page`（从 1 开始）、`size`；任务相关接口响应使用 `items`、`page`、`size`、`totalItems`、`totalPages`；代码托管、通知等模块响应使用 `items`、`page`、`size`、`total`。具体以接口示例为准 |
| 时间格式   | ISO 8601 字符串，如 `yyyy-mm-ddThh:mm:ss+08:00`                                                                                                                              |
| ID 类型  | 后端主键统一使用 `Long`；前端如需 JSON 安全序列化可按字符串展示，但请求参数仍按 `Long` 传递                                                                                                            |
| 文件上传   | MVP 只支持以 `changes[]` 形式上传变更生成 commit，不做网页代码编辑器                                                                                                                          |
| 身份字段   | 涉及当前用户的写操作（接取、提交、审核）一律由后端从 JWT 解析，前端不传 `userId`                                                                                                                       |

## 3. 核心状态枚举

| 类型       | 枚举值                                                                                                  | 说明                                                                                                |
| -------- | ---------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| 用户角色     | `BEGINNER`、`MAINTAINER`、`ADMIN`                                                                       | 后端 / DB 枚举值。前端展示口径为冒险家、委托人、管理员；`VISITOR` 为前端未登录的展示状态，不下发到后端                                  |
| 用户账号状态   | `ACTIVE`、`DISABLED`                                                                                  | DB `users.status` 默认 `ACTIVE`                                                                     |
| 仓库同步     | `PENDING`、`SYNCING`、`SUCCESS`、`FAILED`                                                                | DB `repositories.sync_status` 默认 `PENDING`                                                        |
| Issue 状态 | `OPEN`、`CLOSED`                                                                                       | 由 Gitea/GitHub 同步                                                                                 |
| 悬赏任务状态   | `DRAFT`、`PENDING_ADMIN_REVIEW`、`PUBLISHED`、`IN_PROGRESS`、`IN_REVIEW`、`COMPLETED`、`CLOSED`              | 与 P3 任务与悬赏 / 管理员审核规范一致。`CLOSED` 同时表示管理员下架或任务结束                                                  |
| 接取记录状态   | `ACTIVE`、`COMPLETED`、`CANCELLED`                                                                      | DB `quest_assignments.status` 默认 `ACTIVE`；P4 扩展取消接取后写入 `CANCELLED`                              |
| PR 状态    | `OPEN`、`CHECKING`、`REVIEW_REQUESTED`、`CHANGES_REQUESTED`、`MERGED`、`CLOSED`                            | P3 代码支撑平台接口主要返回 `OPEN`/`MERGED`/`CLOSED`；其余值为 P4 工作台展示扩展，需后端在 PR 同步时映射                       |
| 成果提交状态   | `PENDING_REVIEW`、`APPROVED`、`CHANGES_REQUESTED`、`REJECTED`                                            | DB `submissions.status` 默认 `PENDING_REVIEW`；与 P3 提交评价规范一致                                       |
| 维护者审核决策  | `APPROVED`、`CHANGES_REQUESTED`、`REJECTED`                                                             | 与 P3 提交评价规范 `decision` 取值一致                                                                      |
| 管理员审核决策  | `APPROVE_PUBLISH`、`REJECT_PUBLISH`、`TAKE_DOWN`                                                        | 与 P3 管理员审核规范 `decision` 取值一致                                                                     |

访客说明：未登录访问态由前端维护，主要用于路由、页面展示和权限提示；后端需要鉴权的写操作接口不接受访客操作。访客可浏览公开任务和公开仓库信息，但不能接取任务、提交成果、发布任务、审核成果或执行管理员操作。

## 4. 页面-接口-字段表

### 4.1 登录入口页

| 页面动作 | API | 方法 | 请求字段 | 响应字段 | 错误提示 |
| --- | --- | --- | --- | --- | --- |
| 注册账号 | `/auth/register` | POST | `username`、`email`、`password`、`role`（可选，默认 `BEGINNER`） | `userId`、`username`、`email`、`role`、`status`、`createdAt` | `EMAIL_ALREADY_REGISTERED`、`USERNAME_ALREADY_TAKEN`、`PASSWORD_POLICY_VIOLATION` |
| 登录 | `/auth/login` | POST | `email`、`password`、`remember`（可选） | `accessToken`、`refreshToken`、`tokenType`、`expiresIn`、`user` | `INVALID_CREDENTIALS`、`ACCOUNT_DISABLED` |
| 刷新令牌 | `/auth/refresh` | POST | `refreshToken` | `accessToken`、`tokenType`、`expiresIn` | `TOKEN_EXPIRED`、`UNAUTHORIZED` |
| 登出 | `/auth/logout` | POST | 无（携带 `Authorization`） | 204 无响应体 | `UNAUTHORIZED` |
| 获取当前用户 | `/users/me` | GET | 无 | `userId`、`username`、`email`、`role`、`status`、`createdAt` | `UNAUTHORIZED` |
| 修改密码 | `/users/me/password` | PATCH | `oldPassword`、`newPassword` | `changedAt` | `PASSWORD_POLICY_VIOLATION`、`INVALID_CREDENTIALS` |
| 访客进入 | 前端本地状态 | - | 无 | 前端态：`role=VISITOR` | 访客只读，不允许接取/提交/审核 |

说明：注册、登录、当前用户接口字段均沿用 P3 用户登录与认证规范。前端不要再使用 `username` 作为登录字段，注册时 `email` 必填。

访客权限约束：

- 允许访问：登录入口、悬赏任务板、任务详情、公开仓库/Issue 摘要、前台向导。
- 禁止访问：接取任务、创建分支、上传 commit、发起 PR、提交审核、发布任务、审核成果、管理员上架审核。
- 前端遇到访客点击受限操作时，应提示"请先登录后再执行该操作"。

## 6. 联调优先级

| 优先级 | 接口范围 | 原因 |
| --- | --- | --- |
| 第一批 | 登录、刷新令牌、当前用户、任务列表、任务详情、接取任务 | 前端悬赏任务板和任务详情必须先跑通 |
| 第二批 | 仓库导入、仓库详情、同步、Issue 列表 | 委托人发布任务和工作台依赖仓库上下文 |
| 第三批 | 分支、commit、PR、成果提交 | 冒险家主链路依赖 |
| 第四批 | 维护者审核、管理员审核、异常处理 | 演示闭环和课堂管理员操作依赖 |
| 第五批 | 成长记录、通知、反馈归档 | 轻量成长和持续参与反馈 |
| 第六批 | 推荐、排行榜、徽章 | P2 增强，不阻塞 P0/P1 |

## 7. P4-005 完成判定

本任务完成后应满足：

- 每个核心页面都有对应 API。
- 每个接口明确请求字段、响应字段、状态和错误提示。

- 仓库、Issue、PR、Gitea 集成接口覆盖 MVP 演示链路。
- DB schema（`docs/P4/database/init.sql`）中的字段/状态默认值与本清单一致。
- P3 未定义但 P4 需要的接口（取消接取、同步状态聚合、通知、成长、异常处理、PR 合并、文件浏览等）已标注为 "P4 扩展"，并在后端开发任务中安排补齐规范。
- P2 增强接口已预留，但不阻塞主链路开发。
