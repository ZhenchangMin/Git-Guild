# Git Guild P4 演示说明

> 最后更新：2026-06-07

## 1. 演示目标

本次演示使用本地可运行版本展示 Git Guild 的 P4 核心闭环：

```text
登录 -> 仓库接入 -> Issue 同步 -> 发布悬赏任务 -> 管理员审核上架
-> 冒险家接取 -> Gitea 分支/提交/PR -> 提交成果
-> 维护者审核 -> XP 与成长记录更新
```

演示重点是验证前端、后端、MySQL、Redis、Gitea 之间的集成链路可运行，并能覆盖至少一个正常流程和若干关键异常分支。

## 3. 启动方式

### 3.1 启动基础设施

在仓库根目录执行：

```powershell
cd D:\aaaWW\SoftwareEngineII\project\Git-Guild
docker-compose up -d
docker-compose ps
```

确认 `gitguild-mysql`、`gitguild-redis`、`gitguild-gitea` 均为 `healthy` 或已正常运行。

### 3.2 启动后端

```powershell
cd D:\aaaWW\SoftwareEngineII\project\Git-Guild\backend
.\mvnw.cmd spring-boot:run
```

后端默认读取仓库根目录 `.env`。如果本地没有 `.env`，可以复制 `.env.example` 后按需填写：

```powershell
cd D:\aaaWW\SoftwareEngineII\project\Git-Guild
Copy-Item .env.example .env
```

Gitea 写操作需要有效的 `GITEA_TOKEN`。如果更换了本地 Gitea 管理员或重建了 Gitea volume，需要重新生成 token 并写入 `.env`。

### 3.3 启动前端

```powershell
cd D:\aaaWW\SoftwareEngineII\project\Git-Guild\frontend
npm install
npm run dev
```

Vite 已配置代理：

- `/api` -> `http://localhost:8080`
- `/uploads` -> `http://localhost:8080`

因此浏览器直接访问 `http://localhost:5173` 即可。

## 5. 核心演示路径

### 5.1 登录与角色入口

1. 打开 `http://localhost:5173`。
2. 以游客身份查看任务大厅，确认未登录状态只能浏览公开任务。
3. 使用委托人账号登录，进入公会大厅。
4. 使用管理员账号登录，确认可进入管理员审核台。
5. 使用冒险家账号登录，确认可进入任务大厅、任务详情和冒险家工作台。

预期结果：

- 登录成功后前端保存 session token。
- 受保护路由会按角色跳转或拦截。
- 未登录访问工作台、提交柜台、管理员页面时会跳回登录页。

### 5.2 委托人接入仓库并发布任务

1. 使用委托人账号登录。
2. 进入“仓库接入”或“发布悬赏”相关入口。
3. 填写 Gitea 仓库 URL：

```text
http://localhost:3000/admin-N0vemberRain/gitguild-demo.git
```

4. 点击“导入仓库并使用”。
5. 点击“读取 Issue 列表”，选择 `#1 P4-029 demo issue`。
6. 填写任务信息：

| 字段 | 建议值 |
| --- | --- |
| 标题 | `P4-029 demo issue` |
| 难度 | `B` |
| 技术栈 | `Vue, Spring Boot, Gitea` |
| 预计工时 | `2` |
| 奖励 XP | `80` |
| 完成标准 | 提交 PR，说明修改内容，PR 合并后由维护者审核 |

7. 提交任务。

预期结果：

- 仓库导入后业务库生成或复用 `repositoryId`。
- Issue 同步后可以读取 Gitea Issue。
- 任务创建后状态为 `PENDING_ADMIN_REVIEW`。

### 5.3 管理员审核上架

1. 使用管理员账号登录。
2. 进入管理员控制台的任务审核页。
3. 找到刚发布的任务。
4. 选择审核通过，决策为 `APPROVE_PUBLISH`。

预期结果：

- 任务状态从 `PENDING_ADMIN_REVIEW` 变为 `PUBLISHED`。
- 生成管理员审核记录。
- 冒险家可以在任务大厅看到该任务。

### 5.4 冒险家查看并接取任务

1. 使用冒险家账号登录。
2. 进入悬赏任务板。
3. 搜索或筛选刚发布的任务。
4. 打开任务详情。
5. 点击接取任务。

预期结果：

- 生成 `QuestAssignment` 接取记录，状态为 `ACTIVE`。
- 任务状态从 `PUBLISHED` 变为 `IN_PROGRESS`。
- 返回任务分支名，例如 `task/quest-{questId}-assignment-{assignmentId}-adventurer`。
- 冒险家工作台展示任务、仓库、Issue、PR 和提交入口。

### 5.5 Gitea 分支、提交与 PR

可以在 Gitea UI 或本地 Git 命令中完成代码托管侧操作。建议演示时使用 Gitea UI 更直观：

1. 在 Gitea 仓库中从 `main` 创建任务分支。
2. 在任务分支上修改或新增一个演示文件。
3. 提交 commit。
4. 创建从任务分支到 `main` 的 PR。
5. 回到 Git Guild 冒险家工作台，点击同步 PR 状态。

预期结果：

- Git Guild 能读取 Gitea PR。
- 冒险家工作台能展示 PR 编号、标题和状态。
- “去提交柜台”入口可用。

### 5.6 冒险家提交成果

1. 冒险家进入提交柜台。
2. 选择当前任务。
3. 关联已同步的 PR。
4. 填写成果说明和自检清单。
5. 提交审核。

预期结果：

- 生成 `submissionId`。
- 成果状态为 `PENDING_REVIEW`。
- 任务状态变为 `IN_REVIEW`。
- 维护者审核台可以看到待审核成果。

### 5.7 维护者审核成果

1. 使用委托人账号登录。
2. 进入维护者审核台。
3. 打开待审核成果。
4. 先在 Gitea 中保持 PR 未合并，尝试“审核通过”。

预期结果：

- 后端返回 `PR_NOT_MERGED`。
- 前端明确提示“PR 尚未合并，不能审核通过”。
- 成果仍保持 `PENDING_REVIEW`，任务仍保持 `IN_REVIEW`。

5. 回到 Gitea 合并 PR。
6. 回到 Git Guild 维护者审核台刷新或重新提交审核通过。

预期结果：

- 后端在审核队列读取或审核前同步 Gitea PR 状态。
- PR 状态更新为 `MERGED`。
- 成果状态变为 `APPROVED`。
- 任务状态变为 `COMPLETED`。
- 接取记录状态变为 `COMPLETED`。
- 冒险家获得 XP，成长档案更新。

### 5.8 成长档案与排行榜

1. 使用冒险家账号进入个人档案。
2. 查看 XP、等级、完成任务数、贡献记录。
3. 打开排行榜。

预期结果：

- 冒险家增加本次任务奖励 XP，例如 `80`。
- 完成任务数加 1。
- 贡献记录中出现本次任务完成记录。

## 6. 异常分支演示

| 异常场景            | 操作                                                 | 预期结果                        |
| --------------- | -------------------------------------------------- | --------------------------- |
| 未登录访问受保护页面      | 直接访问 `/workbench`、`/submissions` 或 `/admin/review` | 跳转登录页                       |
| 重复导入同一 Gitea 仓库 | 委托人连续点击“导入仓库并使用”                                   | 复用已有仓库，不再生成新的业务库记录          |
| PR 未合并时审核通过     | 维护者在 PR 仍为 `OPEN` 时点击审核通过                          | 返回并展示 `PR_NOT_MERGED`，状态不推进 |
