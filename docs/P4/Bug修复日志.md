### BUG-001 注册校验失败只提示"请求参数不合法"，未展示后端返回的具体原因

| 字段 | 内容 |

| 是否阻塞演示 | 否 |
| 是否纳入 AI 调试对决 | 是 |

#### 5.1 问题现象

- 触发场景：在登录页"注册"标签提交任何不满足后端校验的输入（最典型：密码填纯数字 `12345678`，前端只校验了 8 位长度、未校验字母+数字混合）。
- 预期行为：表单明确提示是哪条规则没过，例如"password 必须同时包含字母和数字"。
- 实际行为：只弹出笼统的"请求参数不合法"，用户无法判断到底哪里填错，反复试错。
- 影响范围：登录/注册表单所有 `VALIDATION_FAILED` 类错误；因 `httpClient` 为全局共享，凡是经它消费校验错误的页面都同样吞掉了 `details`。
- 复现环境：分支 P4；Chrome；后端 dev profile（MySQL 3307 / Redis 6379 / Gitea 3000，均由 Docker 提供）；admin 种子库。

复现步骤：

1. 打开 http://localhost:5173/ ，点击"注册"。
2. 填用户名 `alice`、邮箱 `alice@example.com`、密码 `12345678`（纯数字）。
3. 提交，观察表单错误提示。

关键日志或截图：

```text
POST /api/v1/auth/register  ->  400
{
  "code": "VALIDATION_FAILED",
  "message": "请求参数不合法",
  "details": "password 必须同时包含字母和数字"
}
前端实际只取了 message 字段展示，details 被丢弃。
```

#### 5.2 根因分析

- 直接原因：`LoginPage.vue` 的 `catch` 分支只读取 `error.message`。
- 深层原因：`httpClient.js` 的 `ApiError` 以 `super(message || code)` 作为 `Error.message`，把后端 `details` 单独挂到 `this.details`；调用方若不主动读取就会丢失。而校验类错误恰恰把"可操作的具体原因"放在 `details`，`message` 只是分类名（"请求参数不合法"）——结构化错误的分层没有在前端消费侧落地。
- 涉及接口或数据表：`POST /api/v1/auth/register`、`POST /api/v1/auth/login`；统一响应体 `ApiResponse{code,message,details}`。无数据表。
- 相关状态流转：无，纯展示层问题。
- 同类隐患排查结果：检索其余使用 `request()` / `ApiError` 的页面，确认 `details` 普遍未被展示；本次先在登录页修复，其它页面登记为后续统一改进点（见 5.5）。

#### 5.3 修复方案

- 修改文件：`frontend/src/pages/auth/LoginPage.vue`
- 关键改动：新增 `resolveAuthError(error)`——当 `error.code === 'VALIDATION_FAILED'` 且存在 `details` 时优先展示 `details`，否则回退 `message`；`catch` 分支改用该函数。业务类错误（如"邮箱已被注册"）的 `message` 本身具体，保持原样。
- 是否影响接口契约：否。后端 `details` 字段早已返回，仅调整前端消费方式。
- 是否涉及数据库迁移：否。
- 是否涉及 Gitea 配置或服务器配置：否。
- 备选方案及未采用原因：曾考虑在 `ApiError` 构造时直接把 `details` 拼进 `message`，但会污染所有调用方，并对 message 本就具体的业务错误产生冗余，故改在展示层按错误类型择优。

#### 5.5 复盘结论

- 经验教训：统一响应体里 `message`（分类）与 `details`（具体原因）的语义分层，必须在前端消费侧真正落地，否则"结构化错误"形同虚设。
- 后续预防措施：在 `httpClient` 层提供统一的 `formatApiError()` 工具并在各页面复用，避免每个页面各写一份错误展示逻辑。
- 是否需要更新 P4-005 接口契约：否。契约已含 `details`；建议补充"前端必须优先展示 details"的使用约定。
- 是否需要更新演示说明：否。
